# D2 Demo Plan: Inductive Types and Type Classes

This document is an instructor-facing plan for the live Lean demos. It does not contain the final Lean code yet. Its purpose is to decide what should be demonstrated, in what order, and what students should notice during each segment.

## Overall Demo Strategy

The live coding should follow the same conceptual thread as the slides:

`inductive -> pattern matching / recursion -> induction -> structure -> common APIs -> class / instance -> integrated example`

The demos should be small, cumulative, and readable. Each demo should introduce one new idea, then immediately show how Lean responds through `#check`, `#eval`, proof states, or `#synth`.

## Suggested Lean File Layout

The final Lean material can either be split across several files or kept in one classroom file. A split layout is better for teaching and maintenance.

Suggested files:

- `code/D2_Inductive.lean`
- `code/D2_PatternMatching.lean`
- `code/D2_InductionProofs.lean`
- `code/D2_Structure.lean`
- `code/D2_CommonTypes.lean`
- `code/D2_Typeclass.lean`
- `code/D2_IntegratedExample.lean`
- `code/D2_Exercises.lean`

If the class needs a simpler setup, these can later be merged into `D2_inductiveType.lean`.

## Demo 0: Warm-Up and Motivation

Estimated time: 5-8 minutes.

Purpose:

- Show that Lean's everyday objects are often inductive.
- Prepare students to think of data as being generated by constructors.
- Establish the main slogan: programs and proofs follow the shape of data.

Live actions:

- Use `#check` on familiar objects such as `Bool`, `Nat`, `Option`, and `List`.
- Point out constructors informally: `true`, `false`, `Nat.zero`, `Nat.succ`, `none`, `some`, `[]`, and `::`.
- Avoid explaining recursors at this stage.

Student observations:

- A type is not just a set of values; Lean knows how values are built.
- Constructor names appear everywhere in Lean programs and proofs.

Instructor questions:

- What are all possible values of `Bool`?
- What are the two possible shapes of an `Option alpha`?
- What are the two possible shapes of a list?

Future Lean material:

- A short warm-up block with `#check` commands.
- No exercises needed here.

## Demo 1: Basic Inductive Types

Estimated time: 12-15 minutes.

Purpose:

- Teach the syntax and meaning of a simple `inductive` declaration.
- Give students a stable checklist for reading any inductive declaration.
- Show that constructors are functions.
- Show the difference between type parameters and constructor arguments.
- Preview the fact that inductive declarations generate elimination principles.

Live actions:

- Present the four questions for reading an inductive declaration:
  what is the new type, what are the constructors, which constructors carry data, and which constructors are recursive?
- Define `Weekday` as a finite closed-world type.
- Use `#check` and `#eval` to inspect constructors and concrete values.
- Define `LoginResult` to show constructors as functions.
- Define `MyOption` to distinguish type parameters from constructor arguments.
- Define `MyNat`, `MyList`, and `Tree` to compare recursive shapes.
- Use `#check Weekday.rec`, `#check MyList.rec`, and `#check Tree.rec` only as a preview.

Student observations:

- Constructors introduce values of the new type.
- Constructors may take arguments.
- Type parameters describe a family of types.
- Recursive constructors refer to the type being defined.
- `deriving Repr` is useful for printing values in demos.
- The recursive shape predicts later recursion and induction.

Instructor questions:

- What is the type being defined?
- Which constructors of this type have no arguments?
- Which constructors carry data?
- Where does recursion appear in the declaration?
- How many branches should a function over this type have?
- For a tree node, how many recursive calls or induction hypotheses should we expect?

Future Lean material:

- `Weekday` as a closed-world finite type.
- `LoginResult` as constructors-as-functions.
- A custom `MyOption`.
- `MyNat` and `MyList`.
- A custom `Tree`.
- Several `#check` commands.
- A few `#eval` examples after deriving `Repr`.
- A short `#check ...rec` block as a preview, not a full recursor lesson.

Possible exercises:

- Add a new constructor to an enumeration type.
- Define a simple `TrafficLight` or `Direction` type.
- Predict the type of each constructor before running `#check`.
- Define a type with one constructor carrying two arguments.
- Identify whether a given inductive declaration is recursive.

## Demo 2: Pattern Matching

Estimated time: 15 minutes.

Purpose:

- Teach pattern matching as elimination by constructors.
- Show how a function can inspect an inductive value.
- Prepare students for recursive functions.

Live actions:

- Define a function on the enumeration type, such as weekend detection.
- Match on `Option` with the `none` and `some` cases.
- Match on lists with the `[]` and `x :: xs` cases.
- Use `#eval` after each definition.

Student observations:

- Each branch corresponds to a possible constructor shape.
- In a constructor branch, Lean exposes the data stored in that constructor.
- Pattern matching is not a trick; it is the natural way to use inductive data.

Instructor questions:

- What cases does Lean force us to consider?
- What information becomes available in the `some` branch?
- What information becomes available in the `x :: xs` branch?

Future Lean material:

- `isWeekend`.
- `getD`-style default extraction from `Option`.
- `safeHead` for lists.
- A few short `#eval` examples.

Possible exercises:

- Write a function that classifies a weekday as workday or weekend.
- Write a function that turns `Option Nat` into `Nat` using a default.
- Write a function that checks whether a list is empty.

## Demo 3: Recursive Definitions

Estimated time: 15 minutes.

Purpose:

- Show how recursion follows the recursive shape of data.
- Introduce structural recursion without going into advanced termination proofs.
- Build definitions that later become induction exercises.

Live actions:

- Define list length recursively.
- Define list map recursively.
- Define list append recursively, or compare with built-in append.
- Optionally define a recursive function on trees, such as size or number of leaves.

Student observations:

- Recursive calls are made on smaller pieces of the input.
- Lean accepts structurally recursive definitions automatically.
- The recursive definition mirrors the data declaration.

Instructor questions:

- Which argument becomes smaller in the recursive call?
- Why is the `[]` case the base case?
- What would go wrong if the function called itself on the same input?

Future Lean material:

- `myLength`.
- `myMap`.
- `myAppend`.
- Optional `Tree.size`.

Possible exercises:

- Define a function that sums a list of natural numbers.
- Define a function that doubles every number in a list.
- Define a recursive function that counts leaves of a tree.

## Demo 4: Proof by Cases

Estimated time: 8-10 minutes.

Purpose:

- Connect pattern matching in programs with case splitting in proofs.
- Show `cases` as the proof analogue of splitting by constructors.

Live actions:

- Prove a simple theorem about `Bool`.
- Prove a simple theorem about `Option`, such as every option is either none or some value.
- Show the proof state before and after `cases`.

Student observations:

- `cases` replaces one goal with one goal per constructor.
- In each branch, Lean knows which constructor was used.
- Non-recursive inductive types usually only need case analysis, not induction.

Instructor questions:

- How many goals should `cases` create?
- What changes in the local context in the `some` branch?
- Why do we not need an induction hypothesis here?

Future Lean material:

- Small examples using `cases`.
- One or two exercises with missing proof terms.

Possible exercises:

- Prove a simple statement by cases on `Bool`.
- Prove a simple statement by cases on `Option alpha`.

## Demo 5: Induction Proofs on Lists

Estimated time: 18-20 minutes.

Purpose:

- Show induction as the proof principle for recursive data.
- Make base case, step case, and induction hypothesis visible.
- Use theorems that connect naturally to the recursive definitions from Demo 3.

Live actions:

- Prove that appending the empty list on the right does not change a list.
- Prove a length theorem for append.
- Prove that mapping the identity function over a list returns the same list.
- Show proof states slowly, especially the induction hypothesis.

Student observations:

- The base case corresponds to `[]`.
- The step case corresponds to `x :: xs`.
- The induction hypothesis is the theorem already known for `xs`.
- Many list proofs become short after `simp` unfolds the relevant definitions.

Instructor questions:

- What is the induction hypothesis saying?
- Why is the theorem for `xs` useful when proving the theorem for `x :: xs`?
- Which part of the proof is computation, and which part uses the induction hypothesis?

Future Lean material:

- A guided theorem about append and empty list.
- A guided theorem about length and append.
- A guided theorem about map and identity.
- Optional challenge theorem involving `Tree.size`.

Possible exercises:

- Fill in the base case.
- Fill in the step case using the induction hypothesis.
- Replace an overly automatic proof with a more explicit one.

## Demo 6: Structures

Estimated time: 15 minutes.

Purpose:

- Explain `structure` as a way to bundle named fields.
- Contrast structures with multi-constructor inductive types.
- Prepare for type classes, which are structure-like objects with instance search.

Live actions:

- Define a simple `Point` structure.
- Construct a point using record syntax.
- Access fields with dot notation.
- Update a field using structure update syntax.
- Define a small environment-like structure to prepare for the integrated expression example.

Student observations:

- A structure has named fields.
- Dot notation is field projection.
- Structure update syntax copies existing data while changing selected fields.
- Structures are common for mathematical objects and bundled data.

Instructor questions:

- What fields are needed to describe this object?
- How is a structure different from an enumeration type?
- Why might an environment be better as a structure than as a bare list?

Future Lean material:

- `Point`.
- `Env` or another small record.
- Field access and update examples.
- Optional theorem showing two structures are equal if their fields agree.

Possible exercises:

- Define a structure for a student record or rectangle.
- Write a function that computes something from the fields.
- Update one field while preserving the others.

## Demo 7: Common Type APIs

Estimated time: 20-25 minutes.

Purpose:

- Introduce common inductive types as tools for modeling different situations.
- Avoid turning the class into an API dump.
- Give one clear use case for each type.

### Option

Purpose:

- Model partiality or failure.

Live actions:

- Define or use a safe lookup/head function.
- Use pattern matching.
- Use default extraction.
- Use `Option.map` if time allows.

Student observations:

- `Option` makes failure explicit in the type.
- Code that consumes `Option` must handle both cases.

### List

Purpose:

- Model finite ordered data.

Live actions:

- Use `map`, `filter`, `foldl` or `foldr`, `length`, and membership.
- Connect the API back to recursive definitions.
- Show one or two simple theorem names, not a long list.

Student observations:

- List APIs package common recursive patterns.
- Theorems about lists often follow the recursive structure of lists.

### Fin

Purpose:

- Model bounded natural numbers and safe indexing.

Live actions:

- Show examples of elements of `Fin 3`.
- Inspect the value and proof fields informally.
- Use `Fin` in a safe indexing scenario if the surrounding code is simple enough.

Student observations:

- `Fin n` is a natural number plus evidence that it is less than `n`.
- Invalid indices are rejected by the type system.

### Multiset

Purpose:

- Model unordered finite data with multiplicity.

Live actions:

- Compare list-like data where order should not matter.
- Show `card`, `count`, or `map`.
- Keep this short unless students already understand quotient-like ideas.

Student observations:

- Lists remember order; multisets do not.
- Multiplicity still matters.
- This portion depends on mathlib.

Instructor questions:

- Which type should we use if lookup can fail?
- Which type should we use if order matters?
- Which type should we use for safe indexing?
- Which type should we use if order should be ignored but repeated elements matter?

Future Lean material:

- One compact block per type.
- A few `#eval` examples.
- Minimal theorem usage.

Possible exercises:

- Use `Option` to handle a failed lookup.
- Use `List.filter` and predict the result.
- Construct an element of `Fin 5`.
- Compare a list example with a multiset example.

## Demo 8: Type Classes and Instances

Estimated time: 20-25 minutes.

Purpose:

- Explain the motivation for type classes.
- Show a class as a structure whose instances Lean can search for.
- Teach the difference between ordinary implicit parameters and type class parameters.
- Introduce `deriving` as a practical tool.

Live actions:

- Define a small type class such as `HasSize`.
- Define instances for `String`, `List alpha`, and a custom type.
- Write a generic function that uses `[HasSize alpha]`.
- Use `#synth` to show instance search.
- Define a custom inductive type with `deriving Repr, BEq, DecidableEq`.
- Show that the derived instances enable printing, comparison, or decidable equality.

Student observations:

- `class` declarations look like structures.
- `instance` declarations register reusable structure.
- Square brackets request instance search.
- `deriving` creates standard instances automatically.

Instructor questions:

- What information does Lean need in order to run this function?
- Where does Lean find the `HasSize` instance?
- What changes if we remove an instance?
- Why is `deriving Repr` useful in live demos?

Future Lean material:

- `HasSize`.
- Several instances.
- `#synth` examples.
- A custom type using `deriving`.
- A short comparison between `{c : C alpha}` and `[C alpha]`.

Possible exercises:

- Add a new `HasSize` instance for a custom structure.
- Predict which instance Lean will synthesize.
- Add a missing `deriving` clause to make a demo evaluate or compare.

## Demo 9: Integrated Expression Example

Estimated time: 10-15 minutes.

Purpose:

- Tie the entire session together in one coherent example.
- Show that the separate ideas are not isolated features.
- Provide a seed for later exercises or homework.

Live actions:

- Define an expression tree with constants, addition, and variables.
- Use pattern matching to define a function over expressions.
- Use `Option` to represent failed variable lookup.
- Use `List` to collect variable names.
- Use a `structure` to represent an environment.
- Use `deriving` to obtain printable and comparable expressions.
- If time allows, prove a very small induction theorem about expressions.

Student observations:

- Syntax trees are naturally inductive.
- Evaluation and analysis are naturally recursive.
- Auxiliary types such as `Option` and `List` appear organically.
- Structures and classes help the example integrate with Lean's ecosystem.

Instructor questions:

- Which constructors make the expression language?
- Which recursive calls are needed for addition?
- Where can variable lookup fail?
- What does the induction hypothesis look like for an expression with two subexpressions?

Future Lean material:

- `Expr`.
- `Env`.
- An evaluator or variable collector.
- A short induction theorem.
- Optional exercises extending the language with multiplication or negation.

Possible exercises:

- Add a new expression constructor.
- Extend the evaluator.
- Collect variables from the new constructor.
- Prove a simple property by induction on expressions.

## In-Class Exercise Rhythm

Recommended pattern:

1. Instructor writes a complete example.
2. Instructor deletes or comments one line and asks students to predict it.
3. Students fill a small hole.
4. Instructor shows the proof state or evaluation result.
5. Move on before the example becomes an API search exercise.

Good exercise types:

- Predict the type of a constructor.
- Complete one branch of a pattern match.
- Fill the base case of an induction proof.
- Identify the induction hypothesis.
- Add an instance and check that `#synth` succeeds.

Avoid during the first pass:

- Complicated termination proofs.
- Heavy quotient explanations for `Multiset`.
- Long theorem search sessions.
- Large tactic scripts that hide the main idea.

## Risk Notes

- `Multiset` depends on mathlib. If setup time is limited, keep it conceptual or use a very short prepared demo.
- `Fin` can become proof-heavy quickly. Keep the focus on safe indexing and the idea of bundled bounds.
- Type classes can feel magical. Use `#synth` and one tiny custom class before mentioning common classes.
- Induction proofs should be short and visible. The goal is to teach the shape of proofs, not to optimize tactic scripts.
